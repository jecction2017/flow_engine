## 1. 引言（Introduction）

### 1.1 编写目的

说明在**告警诊断**场景下，为何需要流程编排类能力，以及本方案在**规则与要素快速变化**约束下的**设计边界、核心机制与成功标准**，供产品、工程、安全运营及评审方对齐。

### 1.2 术语与缩写

| 术语 | 含义 |
|------|------|
| 告警诊断 | 对原始告警进行归一、富集、关联、研判并输出结论/处置建议的过程 |
| 规则 | 分支条件、分级、升级/关闭策略、与下游系统对接逻辑等可变更判定逻辑 |
| 要素 | 告警字段、IOC 类型、情报源、内部字典/映射等随业务演进的输入与参考数据 |
| 编排 | 将多个步骤及其控制流（顺序、并行、循环、失败处理）结构化描述并执行 |
| 草稿 / 版本 | 编辑中定义与已冻结、可引用快照的区分 |
| 通道（生产/灰度） | 已发布定义的不同消费路径，用于渐进放量与验证 |

### 1.3 参考与依赖假设

- 组织已具备告警接入与基础检测能力；本方案聚焦**诊断链路的编排与运营**，不替代检测引擎或全量日志平台。
- 运行环境为**可信内网或私有化**；权限、配额、网关策略由基础设施另行治理。

## 2. 背景与问题陈述（Background & Problem Statement）

### 2.1 业务背景

安全运营需在 SLA 内完成告警处理。诊断链路通常包含：**归一化 → 并行/批量富集 → 汇聚与关联 → 条件研判 → 输出结构化结论**（含证据链字段供工单或 SOAR 消费）。

### 2.2 核心问题

1. **规则高频变更**：分级、自动关闭、升级路径、对接逻辑需经常优化；若绑定在单体代码中，变更周期长、回归成本高。
2. **要素持续演进**：新字段、新 IOC 类型、新情报源、新映射表不断出现；节点间隐式耦合易导致一改多损。
3. **稳定性与时效**：外部源超时或不可用时不应默认拖垮整单；峰值下需可配置的并行与超时策略。
4. **变更风险不可控**：缺乏草稿、版本与灰度时，运营不敢试新规则，问题难以回退与审计。
5. **可观测与可解释不足**：难以快速定位卡在哪一步、难以说明「为何如此判定」，合规与复盘压力大。

### 2.3 问题优先级（建议）

| 优先级 | 问题 | 说明 |
|--------|------|------|
| P0 | 规则/要素变化响应慢 | 直接影响 MTTR 与策略落地 |
| P0 | 变更全量、难回退 | 直接影响生产风险 |
| P1 | 外部源不稳与并行诉求 | 影响 SLA 与资源利用 |
| P1 | 排障与可解释 | 影响运营效率与合规 |
| P2 | 人机边界频繁调整 | 影响协作成本 |

（具体 P 级可由项目组按现状裁剪。）

## 3. 目标与成功标准（Goals & Success Criteria）

### 3.1 业务目标

- 将告警诊断逻辑沉淀为**可独立演进的编排资产**，与应用发版解耦。
- 支持规则与要素变化在**可控风险**下快速验证与推广（草稿 → 版本 → 灰度 → 生产）。
- 在不稳定外部依赖下，通过**可配置的执行与错误策略**保障整体成功率与时效。

### 3.2 非功能目标

- **可追溯**：任意一次执行可关联到所依据的**定义版本**（及通道）。
- **可验证**：变更前可对**单步/单节点**用样本上下文验证。
- **可观测**：执行过程具备结构化状态与分域日志（流程级 / 节点级），并具备合理限额以防失控。
- **可治理**：运行前对编排定义做一致性校验，减少「无效定义上线」。

### 3.3 成功度量（示例指标，由业务取值）

- 规则/要素类需求从需求到可生产使用的**交付周期**下降（基线自定）。
- 因编排错误导致的**生产故障次数/回滚次数**可度量并下降。
- **MTTR** 或「定位到问题节点的时间」改善。
- 灰度期间可统计**误判/漏判**并与版本关联。

## 4. 范围（Scope）

### 4.1 范围内（In Scope）

- 诊断链路的**声明式建模**（任务、循环、子流程、条件、失败策略等）。
- **上下文契约**（节点输入/输出与共享上下文的映射）。
- **数据字典 / 查找表**等与流程解耦的参考数据能力（支撑要素变化）。
- **草稿、版本、多通道发布、实例登记与事件**等生命周期与变更控制。
- **调试与运行观测**（节点级试跑、执行时间线、日志分域与限额）。
- 配套**可视化编排**（与同一套模型对齐，降低 YAML 门槛）。

### 4.2 范围外（Out of Scope）

- 检测规则引擎本体、全量原始日志检索与长期存储产品。
- 面向公网的完整多租户、计费、统一身份体系（可由外围系统承担）。
- 与特定厂商 SOAR/工单系统的**深度专有集成**（可通过适配节点或上下文约定接入，但不作为本说明书必选交付）。

## 6. 方案概述（Solution Overview）

### 6.1 设计思想

采用**「结构层 + 逻辑层 + 生命周期层」**分工：

- **结构层**：诊断步骤与控制流显式声明，规则变更主要落在此处。
- **逻辑层**：单节点内聚的计算与判定，通过**契约**与共享上下文交互。
- **生命周期层**：草稿/版本/通道/实例，使变更**可试、可扩、可回退、可关联**。

### 6.2 概念架构（逻辑视图）

```
[ 告警/要素输入 ] ──► [ 编排运行时：按定义调度步骤与策略 ]
                              │
                              ├──► [ 外部情报/内部服务 ]（可超时/重试/降级）
                              │
                              └──► [ 结构化输出：结论 + 证据链 + 运行轨迹 ]
        ▲
        └── 定义存储：草稿 / 版本 / 通道发布状态
```

### 6.3 与业务问题的映射（摘要）

| 业务问题 | 方案机制 |
|----------|----------|
| 规则难改 | 声明式编排 + 运行前校验 |
| 要素多变、耦合 | 上下文契约 + 字典/查找表与流程分离 |
| 源不稳、要并行 | 可配置执行策略 + 循环/汇聚语义 + 错误策略 |
| 变更风险 | 草稿 / 版本 / 多通道发布 / 实例与事件 |
| 排障与解释 | 节点调试 + 分域日志与结构化运行结果 |

## 7. 详细设计要点（Detailed Design）

### 7.1 编排模型

- **任务节点**：原子诊断步骤，通过脚本或等价逻辑实现，**仅经契约**读写上下文。
- **循环节点**：对 IOC/实体列表展开；支持项隔离策略与向父上下文**汇总结果**，避免并行写冲突。
- **子流程**：复用标准片段（如某类告警的标准富集包）。
- **条件与跳转**：分支显式化；跳转范围受校验约束，防止不可达或非法跳转。
- **执行策略**：为节点或子图绑定超时、并发、重试等策略，区分「必须串行」与「可并行富集」。
- **错误策略**：失败时重试、跳转、继续、自定义处理等，避免默认整单失败。
- **生命周期钩子**：流程级/节点级/循环级前后钩子，用于审计、补偿、打点，与核心业务逻辑解耦。

### 7.2 数据与契约

- **共享上下文**：跨节点传递的统一数据空间（含全局命名约定）。
- **边界（输入/输出映射）**：声明每个节点从上下文读取的路径与写入的路径，作为**稳定契约**。
- **字典与查找表**：承载枚举、映射等**要素类**数据，减少为改表而改控制流。

### 7.3 变更与发布

- **草稿**：编辑中状态，不视为对外承诺版本。
- **版本**：提交后冻结，带版本号与元数据（时间、说明等），供执行与审计引用。
- **通道**：至少区分**生产**与**灰度**（或可扩展）；发布带状态机与超时收敛，避免长期悬挂。
- **实例**：消费者登记所运行版本与通道，支撑「谁在跑哪版」的治理。

### 7.4 调试与观测

- **节点调试**：给定样本上下文，单节点试跑，用于规则/映射快速验证。
- **运行结果**：包含节点级状态、时间线、流程级与节点级日志；日志设条目与长度上限。
- **说明**：调试上下文与完整流程运行的契约行为若有差异，须在用户文档中**明示**，避免误判。

### 7.5 可视化编排

- 与编排模型**同源**，避免「界面一种语义、运行时另一种语义」。
- 支持从列表加载、保存、校验与触发运行（具体交互由产品原型细化）。

## 8. 非功能需求（NFR）

| 类别 | 要求 |
|------|------|
| 可靠性 | 单源失败可降级；执行与日志有上限，防止单次拖垮系统 |
| 可维护性 | 定义可读、可 diff；校验错误信息可定位 |
| 安全 | 可信域部署；敏感数据在上下文与日志中的脱敏策略由组织策略规定 |
| 性能 | 并行富集与超时可配置；峰值行为可观测 |
| 合规 | 版本与执行可追溯 |

## 9. 风险、假设与依赖（Risks, Assumptions, Dependencies）

| 类型 | 内容 |
|------|------|
| 假设 | 使用者具备基本编排与数据契约意识；外围系统可提供稳定接口 |
| 风险 | 过度复杂流程导致难维护 → 需评审模板与子流程复用 |
| 风险 | 灰度样本不足导致误判 → 需配套指标与回滚预案 |
| 依赖 | 情报源与内部服务的 SLA；工单/SOAR 对接契约 |
